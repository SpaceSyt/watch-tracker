# I18n Table Editor Concept

## Goal

Provide a small internal tool for editing translations in an Excel-like table, then saving the result back into `lib/i18n/en.ts` and `lib/i18n/zh.ts`.

## Shape

The tool can start as a local developer utility instead of an app page. A future page can reuse the same parser and serializer.

Table columns:

- `key`: dotted dictionary path, for example `library.renameList`
- `en`: English translation
- `zh`: Simplified Chinese translation
- `notes`: optional context for translators
- `status`: optional marker such as `missing`, `changed`, or `ok`

Rows are generated by flattening the existing dictionary objects. Function entries are shown as read-only expressions until the tool supports parameter-aware editing.

## Workflow

1. Load `lib/i18n/en.ts` as the source schema.
2. Flatten both dictionaries into dotted keys.
3. Render keys in a grid with one language per editable column.
4. Highlight keys that are missing in any supported language.
5. Save by reconstructing nested dictionaries and writing formatted TypeScript files back to `lib/i18n/`.
6. Run `npm run typecheck` to verify both dictionaries still satisfy the shared `Dictionary` type.

## Guardrails

- Do not allow deleting a key from one language only.
- Preserve function-valued entries unless the editor supports typed function templates.
- Keep `en.ts` as the canonical dictionary shape.
- Save generated output in stable key order to keep diffs reviewable.
- Avoid a database for this tool; the source of truth remains `lib/i18n/`.

## First Implementation Option

Build `scripts/i18n-table-editor.ts` with two commands:

- `export`: writes `docs/i18n-table.csv`
- `import`: reads the CSV and updates `lib/i18n/en.ts` / `lib/i18n/zh.ts`

This keeps the first version dependency-free and easy to validate before building a browser UI.
